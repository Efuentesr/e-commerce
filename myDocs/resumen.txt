  ---
  Opción 1 — Railway (la más simple para este proyecto)
  - Soporta Docker Compose directamente
  - PostgreSQL incluido como servicio
  - Free tier disponible
  - Despliegue desde GitHub o CLI

  - Backend Django como Web Service
  - Frontend React como Static Site
  - PostgreSQL administrado
  - Free tier con limitaciones (se "duerme" tras inactividad)

  Opción 3 — Fly.io
  - Muy cercano a Docker, buen control
  - Free tier generoso
  - Requiere algo más de configuración

  ---
  Mi recomendación: Railway

  Es la que menos fricción tiene para un proyecto con docker-compose.yml ya configurado. El mayor trabajo sería:

  1. Separar las variables de entorno para producción (.env.production)
  2. Configurar el VITE_API_URL del frontend para apuntar al backend desplegado
  3. Ajustar ALLOWED_HOSTS y CORS_ALLOWED_ORIGINS en Django
  4. Manejar los archivos de medios (imágenes) — probablemente con un bucket S3 o Cloudflare R2

PWA

  1. Íconos PNG

  Necesitamos al menos dos tamaños:
  - 192×192 — ícono en pantalla de inicio Android
  - 512×512 — splash screen y stores
  - 180×180 — Apple touch icon (iOS)

  Los generaríamos con un script Python usando Pillow (ya instalado en el proyecto): fondo amazon-blue, letra T en
  verde, igual al logo de la app.

  ---
  2. manifest.json en frontend/

  Le dice al navegador que la app es instalable:
  {
    "name": "TiendaOnline",
    "short_name": "Tienda",
    "start_url": "/",
    "display": "standalone",
    "background_color": "#131921",
    "theme_color": "#131921",
    "icons": [...]
  }
  Con "display": "standalone" se abre sin barra de URL, como app nativa.

  ---
  3. Meta tags en index.html

  Para que iOS también lo reconozca (Safari no lee el manifest):
  <link rel="apple-touch-icon" href="/icon-180.png" />
  <meta name="apple-mobile-web-app-capable" content="yes" />
  <meta name="theme-color" content="#131921" />

  ---
  Resultado

  - Android (Chrome): aparece un banner "Instalar app" automáticamente
  - iOS (Safari): el usuario va a Compartir → "Agregar a inicio"
  - En ambos casos: ícono propio, sin barra de URL, pantalla de carga con el color de la app


resumen de lo creado

  frontend/

  Como funciona
  -------------

  ---                                                                                                                     
  1. El navegador detecta que es instalable

  Cuando el navegador carga la app, lee el index.html y encuentra esta línea:

  <link rel="manifest" href="/manifest.json" />

  Va a buscar ese archivo y lee las propiedades. Para mostrar el banner de instalación, Chrome verifica que se cumplan
  estas condiciones:
  - Tiene manifest.json con name, start_url e icons ✅
  - Los íconos incluyen al menos uno de 192×192 ✅
  - El sitio se sirve por HTTPS (en producción) o localhost (en desarrollo) ✅
  - "display": "standalone" está declarado ✅

  Si todo está bien, Chrome muestra automáticamente el banner "Agregar a pantalla de inicio".

  ---
  2. Cómo funciona el ícono en cada sistema

  Android:
  - Usa los íconos declarados en el manifest.json
  - El de 192px aparece en la pantalla de inicio
  - El de 512px se usa como splash screen mientras carga
  - "purpose": "maskable" le permite al sistema recortarlo en círculo, cuadrado redondeado, etc. según el fabricante
  (Samsung, Pixel, etc.)

  iOS:
  - Safari ignora el manifest para los íconos
  - Por eso agregamos esta línea específica en el HTML:
  <link rel="apple-touch-icon" href="/icon-180.png" />
  - 180×180 es el tamaño que usa el iPhone para el ícono en pantalla de inicio

  ---
  3. Qué pasa cuando el usuario la instala

  Al instalar, el sistema crea un acceso directo que abre la app con estas diferencias respecto a abrirla en el
  navegador:

  ┌───────────────────────────────────────┬──────────────────────────────┐
  │          Normal (navegador)           │    Instalada (standalone)    │
  ├───────────────────────────────────────┼──────────────────────────────┤
  │ Se ve la barra de URL                 │ Sin barra de URL             │
  ├───────────────────────────────────────┼──────────────────────────────┤
  │ Se ve la barra de tabs                │ Pantalla completa            │
  ├───────────────────────────────────────┼──────────────────────────────┤
  │ Aparece en el historial del navegador │ Aparece como app separada    │
  ├───────────────────────────────────────┼──────────────────────────────┤
  │ Ícono del navegador en el dock        │ Ícono propio de TiendaOnline │
  └───────────────────────────────────────┴──────────────────────────────┘

  El comportamiento viene de esta línea en el manifest:
  "display": "standalone"

  Cuando se abre en modo standalone, la app arranca directamente en la ruta definida en:
  "start_url": "/"

  El color de la barra de estado del celular (la de arriba con la hora y batería) se toma de:
  "theme_color": "#131921"

  ---
  En resumen

  index.html
    └── carga manifest.json
          └── el navegador verifica condiciones
                └── muestra banner de instalación
                      └── usuario instala
                            └── se abre como app (sin navegador visible)

  El celular simplemente abre Chrome (o Safari) en modo oculto apuntando a tu URL — no es una app nativa real, pero el
  usuario no nota la diferencia para uso cotidiano.


  Plan de despliegue en Railway
  =============================

  Plan de despliegue en Railway

  ---                                                                                                                     La arquitectura final
                                                                                                                          Railway (un proyecto, tres servicios)
  ├── db          → PostgreSQL gestionado por Railway
  ├── backend     → Django + DRF  (tu Dockerfile actual)
  └── frontend    → React + Vite  (tu Dockerfile actual)

  Cada servicio recibe una URL pública automática de Railway.

  ---
  Lo que hay que modificar antes de subir

  1. Backend — docker-compose.yml y settings
  - DEBUG=False en producción
  - ALLOWED_HOSTS con la URL que Railway asigne al backend
  - CORS_ALLOWED_ORIGINS con la URL que Railway asigne al frontend
  - SECRET_KEY real (no la clave de desarrollo)
  - El comando de inicio no debe ejecutar seed_data ni makemigrations cada vez — eso solo se hace una vez

  2. Backend — archivos media
  Las imágenes de productos están en un volumen Docker que Railway no persiste entre deploys. Hay dos opciones:
  - Opción A (simple): ejecutar seed_data una sola vez manualmente después del primer deploy
  - Opción B (robusta): usar Cloudinary para almacenar imágenes — requiere más cambios

  Te recomiendo la Opción A para empezar.

  3. Frontend — VITE_API_URL
  Cambiar de http://192.168.1.33:8000/api a la URL pública del backend en Railway.

  ---
  Los pasos en orden

  1. Crear cuenta en Railway (gratis, con GitHub)
  2. Subir el código a GitHub (si no está ya)
  3. Crear proyecto en Railway y conectar el repo
  4. Configurar los 3 servicios con sus variables de entorno
  5. Primer deploy — Railway construye las imágenes y las levanta
  6. Ejecutar migraciones y seed una sola vez desde la consola de Railway
  7. Crear superusuario desde la consola
  8. Probar con la URL pública

  ---
  Lo más delicado

  El comando de inicio actual en docker-compose.yml hace todo junto:
  makemigrations → migrate → seed_data → runserver

  Para producción lo separamos:
  migrate → runserver   (al iniciar)
  seed_data             (solo una vez, manualmente)

  ---
  Prerequisito

  El código tiene que estar en un repositorio GitHub. ¿Ya lo tenés en GitHub o lo creamos ahora?
